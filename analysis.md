### 6.4 클라이언트 질문 답변

**Q1. 행정 업무 특성상 증거용 이미지나 영상, 대용량 서류가 다수 업로드될 것으로 예상됩니다. 장기적인 관점에서 서버 유지비용을 절감하면서도 데이터 손실 없이 안전하게 대용량 파일을 관리할 수 있는 아키텍처(클라우드 스토리지 활용 등)를 어떻게 설계하실 예정인가요?**

A1. AWS S3 라이프사이클 정책 + Multipart Upload + Presigned URL을 핵심으로 설계합니다.

- 계층형 스토리지 자동 전이 (장기 비용 60~70% 절감)
  - 0~12개월: S3 Standard (행정 진행 중 자료)
  - 12~36개월: S3 Standard-IA (사후 조회용)
  - 36개월 이상: S3 Glacier Instant Retrieval (장기 보관, 분쟁/감사 대비)
  - Lifecycle Rule로 의뢰 종결 시점 + 경과 일자 기준 자동 전이
- 업로드 안정성
  - 멀티파트 업로드(8MB 청크): 네트워크 단절 시 부분 재전송
  - Presigned URL(15분 유효): 클라이언트→S3 직접 업로드로 백엔드 부하 제거
  - 업로드 완료 시 Lambda 트리거로 ETag/SHA-256 무결성 검증 + DB 메타 업데이트
- 데이터 손실 방지
  - 버전 관리(Versioning) 활성화
  - 서울↔도쿄 Cross-Region Replication
  - 백업 보관 기간 7년 (개인정보보호법 권고치)
- 보안
  - SSE-KMS 서버측 암호화 + 접근 시 서명 URL(만료 5분)
  - IAM 정책: 본인 파일만 조회, 행정사는 견적 제출한 의뢰 첨부만 조회, 관리자는 감사 로그 조건부 접근
  - Lambda + ClamAV 바이러스 스캔 → 감염 시 격리 버킷 이동
- 모니터링
  - CloudWatch + S3 Storage Lens 용량/비용 대시보드
  - 월간 비용 알림 임계치 설정

**Q2. 카카오 알림톡 기반 실시간 대응 체계 — 행정사가 앱을 계속 켜두지 않아도 의뢰인의 요청을 즉시 인지하고 대응할 수 있도록, 카카오 알림톡 API와 웹 서버를 어떻게 연동할 계획입니까? 특히 알림톡 내 버튼을 클릭했을 때 특정 견적서 페이지로 바로 연결되는 기능(딥링크)을 어떻게 구현하시겠습니까?**

A2. 카카오 알림톡 BizMessage API + SQS 비동기 큐 + 서명 토큰 기반 딥링크로 구성합니다.

- 발송 트리거 흐름
  1. 의뢰인 의뢰서 등록 → 백엔드 매칭 엔진이 카테고리 일치 행정사 후보 추출
  2. 후보별 SQS 큐에 발송 작업 적재 (Fan-out)
  3. Lambda Consumer가 카카오 BizMessage API (NHN/알리고/Solapi 등) 호출
  4. 행정사는 카카오톡 푸시로 즉시 인지 → 버튼 클릭으로 즉시 진입
- 알림톡 템플릿 (사전 심사 승인 필요)
  - 변수: `#{행정사명}`, `#{의뢰카테고리}`, `#{의뢰요약}`, `#{마감일}`
  - 버튼: "견적 작성하기" → 딥링크
- 딥링크 구현
  - URL: `https://{도메인}/scrivener/requests/{requestId}?token={signedToken}`
  - signedToken: HMAC-SHA256, 페이로드 `{ scrivenerId, requestId, exp(24h) }`
  - 로그인 상태면 즉시 진입 / 미로그인이면 카카오 SSO 자동 로그인 후 진입 (one-tap)
  - 만료/위조 토큰은 안전 페이지로 리다이렉트
  - 1회용 옵션(Redis nonce) 추가 가능
- 신뢰성 (Fallback)
  - SQS Dead Letter Queue + 지수 백오프 재시도 3회
  - 최종 실패 시 LMS/SMS 자동 폴백 (동일 딥링크 포함)
  - 웹 Push API 병행 발송
- 운영 가시성
  - 관리자 대시보드에서 발송/도달/클릭률 통계 (Chart.js)
  - 알림톡 템플릿 트리거 ON/OFF + 변수 검증 UI
  - 발송 로그 90일 보관 + S3 아카이브 1년
